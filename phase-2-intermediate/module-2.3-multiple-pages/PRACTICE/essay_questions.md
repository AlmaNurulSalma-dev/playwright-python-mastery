## Question 1 (Easy)

Jelaskan perbedaan antara membuka tab baru dengan `context.new_page()`
dan menangkap popup dengan `page.expect_popup()`.

a. Kapan kamu pakai `context.new_page()` dan kapan pakai `expect_popup()`?
b. Kenapa Playwright TIDAK butuh "switch to window" seperti Selenium?
   Berikan contoh code comparison.
c. Apa yang terjadi dengan cookies kalau kamu buka tab baru
   di context yang SAMA vs context yang BERBEDA?

### Answer:
<!-- Tulis jawaban kamu di sini -->

a. Perbedaan antara context.new_page() dan page.expect_popup() terletak pada timing dan bagaimana kamu mengontrol pembukaan window. context.new_page() digunakan ketika kamu secara intentional dan deliberately ingin membuka tab baru untuk testing multiple pages dalam context yang sama. Ini dipakai ketika kamu punya control penuh terhadap waktu kapan tab dibuka, misalnya saat kamu click button yang akan membuka tab baru atau ketika kamu secara programmatic memerlukan multiple pages berjalan simultaneous. Contoh kasus penggunaan context.new_page() adalah testing collaborative features, membuka product di tab yang berbeda untuk compare harga, atau testing workflow dimana user perlu interact dengan multiple pages secara bersamaan. Sebaliknya, page.expect_popup() dipakai ketika ada popup window yang akan muncul sebagai hasil dari user action seperti click, form submit, atau navigation. Popup bersifat unpredictable atau tidak direct dalam arti bahwa kamu trigger action dulu, kemudian window muncul sebagai hasilnya. Kamu perlu capture dan handle popup tersebut sebelum atau bersamaan dengan action selesai. Contoh kasus adalah print dialog yang muncul setelah user click print button, OAuth login popup yang muncul ketika user click login with Google, confirmation windows yang muncul setelah action tertentu, atau cross-origin popups yang dibuka oleh website target. Jadi key difference-nya adalah new_page() adalah deliberate controlled opening dimana kamu inisiasi sendiri, sementara expect_popup() adalah reactive handling dari popup yang muncul sebagai result dari action tertentu.

b. Playwright tidak butuh manual "switch to window" seperti Selenium karena fundamental architecture-nya yang berbeda. Dalam Selenium, browser driver menyimpan global state tentang window mana yang currently active. Ketika popup muncul, Selenium tidak langsung aware tentang popup tersebut, sehingga kamu harus manually get semua window handles menggunakan driver.window_handles, identify mana yang window baru, kemudian switch to window tersebut menggunakan driver.switch_to.window(handle). Ini berarti untuk interact dengan popup, kamu harus switch context dulu, lakukan actions, kemudian switch back ke original window. Proses ini menambah complexity dan error-prone karena kamu harus track window handles sebagai strings dan manage global "current window" state. Sebaliknya, Playwright menggunakan direct object reference architecture dimana setiap page atau popup adalah object terpisah yang bisa kamu interact langsung tanpa perlu switch context. Ketika popup muncul, kamu bisa capture popup reference menggunakan waitForEvent('popup') dan langsung interact dengan popup object tersebut. Contoh dengan Selenium, ketika kamu click button yang membuka popup, kamu harus melakukan: driver.find_element("xpath", "//button").click() untuk trigger action, kemudian time.sleep(1) untuk wait popup muncul, then driver.switch_to.window(driver.window_handles[1]) untuk switch ke popup, lakukan actions di popup, kemudian driver.switch_to.window(driver.window_handles[0]) untuk switch back ke original window. Ini memerlukan banyak boilerplate code dan management yang cumbersome. Dengan Playwright, kamu bisa melakukan: const [popup] = await Promise.all([page.waitForEvent('popup'), page.click('button')]). Kemudian kamu bisa langsung interact dengan popup reference: await popup.fill('#input', 'data') tanpa perlu switch context. Kamu bahkan bisa interact dengan original page tanpa switch: await page.fill('#field', 'value'). Ini lebih powerful karena Playwright allows simultaneous access ke multiple pages without context switching. Alasan teknis mengapa Playwright lebih baik adalah karena setiap page memiliki isolated API context, sehingga tidak ada global state tracking. Direct object reference juga membuat code lebih type-safe dengan IDE support dan autocomplete. Promise-based approach naturally handles async popup creation dengan clean syntax. Dengan Selenium, kamu forced untuk manage window handles sebagai global mutable state, yang membuat code lebih fragile dan harder to maintain.

c. Behavior cookies sangat berbeda tergantung apakah tab dibuka di context yang sama atau context yang berbeda. Ketika kamu membuka tab baru menggunakan context.new_page() di context yang sama, semua cookies, localStorage, sessionStorage, dan semua browser storage akan di-share antar semua pages dalam context tersebut. Ini berarti jika Page 1 melakukan login dan mendapatkan session cookie dari server, Page 2 yang dibuat dalam context yang sama akan immediately bisa access protected content tanpa perlu login lagi karena cookies sudah di-share di level context. Contohnya, kalau kamu punya authenticated page dengan session_id cookie, membuka page baru di context yang sama akan automatically include session_id cookie di semua HTTP requests dari page baru itu, sehingga user sudah considered logged in tanpa explicit login flow. Ini sangat useful untuk testing multi-page workflows dimana authentication state harus persist across pages, atau testing collaborative features dimana multiple pages harus share session context. Sebaliknya, ketika kamu membuat page menggunakan browser.newContext() untuk create context yang completely berbeda, setiap context memiliki isolated dan completely separate cookies, localStorage, dan sessionStorage. Context yang berbeda adalah like private browsing windows yang tidak share state satu sama lain. Jadi kalau Page 1 login di Context 1 dan mendapatkan session_id cookie, Page 2 di Context 2 tidak akan punya akses sama sekali ke cookies dari Context 1. Ketika Page 2 try access protected endpoint, server akan return 401 Unauthorized atau redirect ke login page karena session_id cookie tidak ada, dan user harus login ulang di Context 2. Ini useful untuk testing multi-user scenarios dimana kamu perlu completely separate authentication states yang tidak leak ke each other, seperti testing admin user dan customer user yang harus fully isolated, atau testing permission systems dimana one user TIDAK boleh punya akses ke another user's data. Kesimpulannya, same context equals shared cookies dan authentication state yang persists across all pages in context, sementara different contexts equals completely isolated state seperti separate users atau private browsing sessions. Kalau testing authentication dan authorization features, memahami ini adalah critical karena context boundary adalah dimana authentication state gets isolated.

## Question 2 (Medium)

Tentang session state management:

a. Jelaskan apa yang disimpan oleh `context.storage_state(path="state.json")`.
   Apa saja isi file JSON tersebut?
b. Berikan 3 real-world use case dimana menyimpan dan me-reuse
   session state sangat berguna.
c. Kamu punya scraper yang harus login ke website setiap kali jalan.
   Login butuh 5 detik (fill form, click, wait redirect).
   Scraper jalan 100 kali sehari. Berapa waktu yang bisa kamu hemat
   dengan session state? Jelaskan approach-nya.

### Answer:
<!-- Tulis jawaban kamu di sini -->

a. Method context.storageState(path="state.json") digunakan untuk export semua authentication dan browser storage data dari context ke dalam file JSON yang bisa di-reuse di kemudian hari. File JSON yang dihasilkan memiliki struktur yang terdiri dari dua bagian utama yaitu cookies dan origins. Bagian cookies adalah array yang menyimpan semua cookies beserta metadata lengkapnya. Setiap cookie object di dalamnya contains properties seperti name yang adalah identifier dari cookie, value yang adalah actual data yang disimpan, domain yang adalah domain dimana cookie berlaku, path yang menspesifikasi path cookie, expires yang adalah timestamp expiry, dan boolean flags seperti httpOnly yang indicates apakah cookie accessible dari JavaScript, dan secure yang indicates apakah cookie hanya dikirim over HTTPS. Contohnya, setelah user login, server mungkin set session_id cookie dengan httpOnly=true dan secure=true, atau auth_token dengan value berupa JWT token. Semua cookies ini akan di-capture dalam storage state JSON. Bagian origins adalah array yang menyimpan localStorage dan sessionStorage untuk setiap origin domain. Setiap origin object memiliki origin field yang adalah domain URL seperti "https://example.com", dan dua nested arrays yaitu localStorage dan sessionStorage. localStorage menyimpan persistent data yang persist across browser sessions, seperti user preferences dimana user set theme=dark dan language=id, atau last_visited yang menyimpan last page user visited. sessionStorage menyimpan temporary session-specific data yang only exists untuk current session, seperti temp_filter yang menyimpan current filter yang user apply saat browsing. Penting untuk diingat bahwa storageState() HANYA menyimpan cookies, localStorage, dan sessionStorage. Method ini TIDAK menyimpan data yang disimpan di IndexedDB yang digunakan oleh modern web apps untuk offline functionality, TIDAK menyimpan WebSQL data, TIDAK menyimpan Service Worker state yang diperlukan untuk offline support, dan TIDAK menyimpan Cache API data. Jadi untuk aplikasi yang heavily rely pada IndexedDB atau Service Workers, storageState() mungkin tidak cukup untuk fully replicate state dan user mungkin perlu perform additional setup steps. Tetapi untuk mayoritas web applications yang menggunakan traditional authentication dengan cookies dan localStorage, storageState() usually sufficient untuk capture semua necessary state untuk session reuse.

b. Use case pertama adalah automated testing suites yang login-heavy dimana kamu punya many test cases yang semuanya memerlukan authenticated user. Dalam typical e-commerce testing, mungkin kamu punya 20 atau lebih test cases yang semuanya test different features seperti product search, add to cart, checkout flow, user profile update, wishlist management, dan sebagainya. Jika setiap test case harus perform login dari scratch, dan login process butuh 5-10 detik untuk fill form, submit, dan wait for redirect, maka total waktu spent just on authentication bisa mencapai 100 detik atau lebih. Dengan menyimpan session state setelah initial login, kamu bisa reuse session tersebut untuk semua subsequent test cases dengan simply load previous state menggunakan browser.newContext({ storageState: 'saved_state.json' }). Ini memungkinkan test suite jalan 5x lebih cepat, sehingga developers bisa mendapatkan faster feedback loop during development, dan automated test suites bisa complete dalam reasonable time. Semua waktu yang saved dari authentication skips bisa diused untuk actual feature testing. Use case kedua adalah web scraping dengan pagination dan session management, terutama untuk scrapers yang target protected resources yang memerlukan authentication. Bayangkan kamu develop scraper untuk product inventory di e-commerce site, dan scraper harus extract thousands of items across many pages. Login butuh 5 detik, but scraper punya thousands of pages to scrape. Jika scraper di-run multiple times sehari dan session timeout atau scraper crash di middle of pagination, kamu harus start dari beginning lagi. Dengan strategy saving session state di checkpoints, misal setelah scrape setiap 50 items, scraper bisa track progress dan auto-resume dari last checkpoint jika terjadi failure. Ini making scraper much more robust dan resilient terhadap temporary network issues, server errors, atau crashes. Scraper bisa run unattended overnight tanpa fear kehilangan progress. Use case ketiga adalah data seeding untuk demo atau load testing environments. Bayangkan kamu perlu setup demo environment dengan 1000 simulated users yang semuanya authenticated. Jika kamu create users one by one dengan sequential login, dan login butuh 5 detik per user, maka total setup time adalah 5000 detik atau 83 minutes just untuk authentication. Dengan smarter approach, kamu bisa create batch of users (misal 5 users), perform authentication untuk each, save their session states. Kemudian kamu bisa programmatically clone atau replicate these session states untuk total 1000 users tanpa actually running login sequence 1000 times. Setup selesai dalam 10 menit instead of 83 minutes, dan environment jadi replicatable dan consistent. Ini especially valuable ketika preparing demo untuk stakeholders atau setting up environment untuk load testing tools yang expect pre-authenticated sessions.

c. Kalau scraper jalan 100 kali sehari dan setiap login memakan 5 detik, maka total waktu terbuang hanya untuk login adalah 500 detik atau 8.33 menit per hari. Dalam setahun dengan asumsi scraper jalan 250 hari working days, total waktu terbuang adalah sekitar 2000 menit atau lebih dari 33 jam. Ini significant amount of time yang bisa dihemat. Approach untuk mengoptimasi ini menggunakan session state caching dan intelligent state reuse adalah sebagai berikut. Pertama, kamu define session expiry window, misal 24 jam. Pada initialization, sebelum melakukan scraping, scraper check apakah file session state sudah exist dan apakah masih valid berdasarkan file timestamp. Jika file tidak exist atau sudah expired, scraper perform login normal dengan fill credentials, submit form, wait untuk navigation, dan save session state ke file menggunakan await context.storageState({ path: './session_state.json' }). Setelah session state disave, scraper proceed dengan actual scraping. Pada subsequent runs hari yang sama, scraper load existing session state menggunakan await browser.newContext({ storageState: './session_state.json' }) dan navigate langsung ke scraping target tanpa login. Scraping request otomatis include session cookies dari loaded state, sehingga server recognize user sebagai authenticated. Ini skip seluruh 5 detik login overhead. Dengan 100 runs per hari, overhead menjadi hanya 5 detik untuk one initial login plus roughly 0.5 detik untuk each scraping request tanpa login, total 50 detik untuk 100 scrapes, making total daily overhead hanya 55 detik. Dibanding dengan 500 detik sebelumnya, itu saving 7.4 menit per hari atau lebih dari 50 jam per tahun. Approach ini juga memberikan benefits tambahan seperti significantly less server load karena fewer authentication requests yang reduces strain pada authentication service, faster feedback loop during scraper development karena skip login delays, dan more resilient scraper karena bisa retry dengan same session. Code untuk implement ini involve checking file existence dan modification time, loading state conditionally, dan adding error handling untuk detect session expiry. Jika scraper encounters error ketika trying access protected resource dengan cached session, seperti server returns 401 Unauthorized atau 403 Forbidden, itu indikasi bahwa session sudah expired atau invalidated by server. Scraper bisa detect error codes dan automatically perform fresh login, delete old state file, save new state, dan retry operation. Ini making scraper intelligent tentang session management dan robust terhadap unexpected session expiry atau server-side state changes. Dengan kombinasi session state reuse dan error recovery, scraper jadi efficient, resilient, dan praktis untuk production use.

## Question 3 (Hard)

Kamu diminta membuat automation tool untuk testing sebuah e-commerce website.
Requirements:
- Test login sebagai Admin (full access) dan Customer (limited access)
- Admin harus bisa akses Dashboard + User Management
- Customer hanya bisa akses Product Catalog + Cart
- Kedua user harus jalan BERSAMAAN (tidak sequential)
- Harus verify bahwa Customer TIDAK bisa akses Dashboard

a. Gambarkan architecture yang kamu akan buat:
   - Berapa browser instances?
   - Berapa contexts?
   - Berapa pages per context?
   Gunakan tree format.

b. Tulis pseudocode untuk seluruh flow, termasuk:
   - Login kedua user
   - Navigate ke pages masing-masing
   - Verify access controls
   - Save sessions

c. Kenapa menggunakan contexts lebih baik dibanding 2 browser instances
   untuk scenario ini? Sebutkan 3 alasan.

### Answer:
<!-- Tulis jawaban kamu di sini -->

a. Architecture yang saya design menggunakan single Browser instance yang shared, di dalamnya ada two completely isolated Contexts, dan each context punya multiple Pages untuk handle different parts of user journey. Browser instance adalah single Chromium process yang di-launch sekali pada start of test. Di dalamnya terdapat Context 1 yang dedicated untuk Admin user, ini adalah isolated environment dengan separate cookies dan storage dimana admin authentication state exists. Context 1 memiliki tiga Pages yang masing-masing serve different purpose. Page 1 adalah Login Page dengan URL /login yang digunakan untuk perform initial authentication flow, handling email dan password field filling, form submission, dan waiting untuk navigation ke authenticated area. Page 2 adalah Dashboard Page dengan URL /dashboard yang adalah admin-only area untuk viewing analytics, business metrics, revenue charts, dan key performance indicators. Page 3 adalah User Management Page dengan URL /admin/users yang adalah admin-only area untuk viewing list of users dalam sistem, creating new users, editing user details, deleting users, dan managing permissions. Semua Pages dalam Context 1 share same cookies dan localStorage, sehingga authentication state persist across semua Pages. Context 2 adalah completely isolated context untuk Customer user dengan separate credentials dan completely isolated cookies. Context 2 juga memiliki multiple Pages. Page 1 adalah Login Page dengan URL /login untuk customer authentication flow, handling form filling dengan customer credentials. Page 2 adalah Product Catalog Page dengan URL /products untuk browsing available products, searching products, filtering by category, dan viewing product details. Page 3 adalah Shopping Cart Page dengan URL /cart untuk viewing items dalam cart, modifying quantities, applying discount codes, dan proceeding ke checkout. Page 4 adalah Dashboard Page dengan URL /dashboard yang digunakan specifically untuk security testing, to verify bahwa customer TIDAK bisa access admin dashboard. Ketika customer try navigate ke dashboard, expectation adalah customer receive 403 Forbidden response atau automatic redirect ke home page atau access denied message. Tree structure-nya:
Browser Instance (Chrome)
├─ Context 1: Admin (admin@example.com)
│  ├─ Page 1: Login (/login)
│  ├─ Page 2: Dashboard (/dashboard)
│  └─ Page 3: User Management (/admin/users)
│
└─ Context 2: Customer (customer@example.com)
   ├─ Page 1: Login (/login)
   ├─ Page 2: Product Catalog (/products)
   ├─ Page 3: Shopping Cart (/cart)
   └─ Page 4: Dashboard (/dashboard) [Security Test]
Architecture ini menggunakan satu Browser instance dan dua Contexts dengan alasan efficiency. Single Browser instance berarti single Chromium rendering engine, single JavaScript execution engine, single GPU process, sehingga resource sharing dan memory efficiency maksimal. Jika menggunakan two browser instances, setiap instance would require full Chromium startup dengan semua engines dan processes, total memory usage would be roughly 360MB. Dengan single browser dan two contexts, total memory hanya around 240MB, saving 33% memory. Multiple Pages per context allow us test complex user journeys dan verify state persistence, while two Contexts ensure complete isolation antara admin dan customer authentication states sehingga cookies TIDAK leak, dan testing security restrictions becomes practical.

b. Test flow dimulai dengan setup phase dimana browser di-launch dan two contexts diciptakan dengan same viewport dan configuration untuk consistent testing environment. Setup mencakup launching Chromium menggunakan chromium.launch(), creating admin context menggunakan browser.newContext() dengan viewport 1280x720, creating customer context dengan same configuration, dan creating pages untuk each context. Setelah setup complete, kedua user logins dimulai secara parallel menggunakan Promise.all() untuk maximize efficiency. Admin login flow melibatkan navigating admin Page 1 ke login URL, waiting untuk login form selector, filling email field dengan admin@example.com, filling password field dengan admin password, clicking submit button, dan waiting untuk navigation ke dashboard atau authenticated area. Setelah successful navigation, we save admin session state menggunakan adminContext.storageState({ path: './admin_session.json' }). Parallel dengan admin login, customer login flow berjalan di customer context dengan same steps tetapi customer credentials. Important thing di customer login adalah after login succeeds, kita verify bahwa customer di-redirect ke /products atau /home page, BUKAN ke /dashboard. Jika customer di-redirect ke dashboard, itu already security issue. Setelah both users logged in successfully, kita proceed dengan parallel testing untuk admin dan customer. Admin tests involve two main flows. Dashboard test navigates admin Page 2 ke /dashboard, waits untuk dashboard elements untuk load seperti analytics widget, revenue chart, top selling products widget. We query page untuk semua admin-only elements menggunakan page.$() atau page.$$() selectors, dan assert bahwa elements exist dan contain expected data. We also wait untuk charts untuk render. User Management test navigates admin Page 3 ke /admin/users, waits untuk user table untuk load, counts jumlah rows di table, tests search functionality dengan fill search input dengan customer email, waits untuk filter debounce, dan verify filtered results returned correct users. Customer tests involve product catalog dan cart flows. Catalog test navigates customer Page 2 ke /products, waits untuk product cards untuk load, counts products, performs search dengan keyword 'laptop', waits untuk filter, verifies filtered products returned. Cart test involves finding first product add-to-cart button, clicking it, listening untuk API response, waiting untuk cart count badge untuk update to '1', then navigating customer Page 3 ke /cart, waiting untuk cart items untuk load, verifying items display dengan correct quantities dan prices, dan verifying total calculated correctly. Security tests adalah critical part yang tests access control boundaries. Dashboard security test navigates customer Page 4 ke /dashboard dengan customer cookies, waits untuk page load, then checks final URL. Expected behavior adalah either page redirects customer back to /home or /products, atau page shows 403 Forbidden error, atau page displays "Access Denied" message. We verify menggunakan boolean checks on final URL dan page content. If customer successfully loads /dashboard, itu security breach dan test fails. User Management security test performs similar check dengan navigating ke /admin/users dengan customer context, expecting 403 or redirect. After all tests complete, session states disave ke files untuk potential reuse di test runs berikutnya. Cleanup phase closes both contexts dan browser ke free resources. Jika any test fails during parallel execution, entire test fails dengan error message specifying which test failed. Test output provides clear pass/fail status untuk each area.

c. Alasan pertama adalah resource efficiency dan memory conservation. Setiap browser instance memerlukan launch full Chromium process yang includes rendering engine, JavaScript execution engine, GPU process, DOM parser, dan many internal background processes untuk handle various browser functionalities. Total memory requirement per browser instance adalah roughly 180MB, meaning dua browser instances menggunakan ~360MB total. Ketika testing dengan multiple users, jika kamu scale ke 10 users dengan dua instances per user, itu 20 instances total consuming ~3.6GB memory. Sebaliknya, single browser instance dengan multiple contexts memerlukan single Chromium engine launch (~200MB) plus lightweight context wrappers (~20MB each). Untuk 10 users dengan single browser, memory usage adalah roughly 200MB plus 10x20MB contexts equals ~400MB total. Memory savings bisa 80-90% depending on scale. Memory yang disaved ini valuable untuk either running more tests concurrently atau reducing infrastructure costs ketika running tests pada cloud environments where compute charged per memory usage. Memory efficiency juga important ketika running test suites on resource-constrained devices atau CI/CD pipelines yang run many test suites simultaneously.
Alasan kedua adalah startup performance dan time efficiency. Browser startup adalah expensive operation. Launching Chromium process dari zero memakan 3-5 detik because perlu start operating system process, initialize rendering engine, allocate memory, establish IPC communication channels antara parent dan child processes, dan setup GPU acceleration. Dengan dua browser instances, startup time adalah roughly 3-5 seconds untuk first browser plus 3-5 seconds untuk second browser equals ~7 seconds total. Dengan single browser plus two contexts, startup time adalah 3-5 seconds untuk initial browser launch plus ~0.5 seconds per context creation equals ~5.5 seconds total. Time saved ~1.5 seconds per test run. Untuk test suites yang run 100 times daily, itu saving ~2.5 minutes daily atau ~625 minutes yearly. Faster startup time berarti faster feedback loop untuk developers during development, faster test runs untuk CI/CD pipelines meaning faster merge velocity untuk development team, dan more efficient resource utilization.
Alasan ketiga adalah synchronization simplicity dan error handling clarity. Dengan two browser instances, kamu punya two completely separate processes yang TIDAK share event loop atau state. Jika kamu try synchronize operations across instances, misalnya "ensure both users logged in BEFORE proceeding dengan tests", kamu harus manage separate error paths untuk each instance. Jika admin login succeeds tetapi customer login fails, kamu harus handle incomplete state, cleanup resources, dan report meaningful error. Code menjadi complex dengan nested error handlers dan manual state tracking. Dengan single browser dan two contexts, semua contexts run dalam single Node.js process dengan shared event loop. Synchronization menjadi trivial menggunakan Promise.all([adminLogin(), customerLogin()]). Jika salah satu login fails, Promise.all atomically rejects, entire test fails immediately dengan clear error, dan cleanup straightforward. Code menjadi simpler dan more maintainable. Error messages lebih meaningful karena single execution context memastikan deterministic behavior. Debugging jadi easier karena kamu inspect single browser state instead of juggling multiple browser debugging protocols. Testing parallelism natural dengan Promise.all tanpa need untuk complex coordination mechanisms. Single browser reference dalam code juga makes code cleaner dan easier untuk understand karena no need untuk pass multiple browser instances through test functions, reducing parameter passing dan improving readability.
